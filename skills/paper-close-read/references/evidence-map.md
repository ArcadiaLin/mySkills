# Evidence Map Guide

## Goal

在写 `analysis.md` 之前，先把论文拆成“claim -> evidence -> source location”的地图，避免边看边写时丢数字、丢图号、误判论证强度。

## Minimum Evidence Map Contents

对复杂、定量、图表密集或方法密集论文，默认应整理出这些内容；可以写成单独的 `evidence_map.md`，也可以先作为明确的草稿：

### 1. 基本定位

- 标题
- 作者
- 期刊 / 会议
- DOI 或本地标识
- 论文类型：方法 / 实验 / 实证 / 计量 / 历史 / 综述 / 理论

### 2. 章节定位

- 摘要在什么位置
- 引言 / 背景在什么位置
- 方法 / 分析设计在什么位置
- 结果 / 实证分析在什么位置
- 讨论 / 结论在什么位置

### 3. Claim-Evidence 表

至少整理出 3 到 8 条核心 claim，每条都回答：

- 作者的 claim 是什么
- 对应证据是什么
- 证据来自哪个图、表、公式、页码或段落
- 最关键的数字是什么
- 证据强度如何
- 这条证据在正文里准备放在哪一节

示例：

```markdown
| claim | source | key numbers | evidence type | notes |
| --- | --- | --- | --- | --- |
| GS 几乎是 WoS/Scopus 的超集 | Fig. 3, Table 2 | 95%, 92%, 94% | overlap + coverage | 可放在“主实验结果” |
| 20 世纪后期增长率显著抬升 | Table 2, Fig. 2 | 8.13%, doubling 8.9 years | segmented regression | 可放在“主结果” |
```

### 4. 公式与符号表

如果论文有公式、模型、回归、目标函数、定义式或统计量，至少记录：

- 公式编号或位置
- 变量和符号含义
- 公式在全文中的作用
- 写正文时需要解释的直觉

### 5. 图表使用计划

至少记录：

- 哪张图用于方法解释
- 哪张图用于主结果
- 哪张图用于补充机制、消融、案例或稳健性
- 哪些图表虽然存在但不准备用，以及原因

### 6. 关键数字摘录

把正文要反复使用的数字先摘出来，避免写作时来回翻找或写错：

- 样本量
- 时间范围
- 数据集规模
- 最关键参数
- 主结果数值
- 效应量 / 增长率 / 翻倍时间 / 显著性

## Extraction Rules

- 优先摘“会进入正文”的数字，而不是机械抄全表。
- 所有关键数字都尽量带来源，例如 `Table 2`、`Fig. 4`、`Eq. (1)`、`page 5`。
- 如果只能从正文段落中读出数字，也要记录页码或段落位置。
- 如果数字因 OCR 或提取噪声不完全确定，标注“不确定”，不要悄悄猜。

## When to Create a File

满足任一条件时，优先在输出目录里显式创建 `evidence_map.md`：

- 论文超过 10 页
- 图表较多
- 定量结果较多
- 公式较多
- 同时存在多个子问题、多个数据源或多个实验部分

如果论文很短，也至少要按同样结构在草稿中完成证据整理，再开始写。

## Pre-Draft Checks

开始写 `analysis.md` 前，确认这些问题都能回答：

- 论文的核心 claim 是哪几条
- 每条 claim 的证据图表是哪几个
- 最重要的数字是什么
- 有无摘要原文
- 有无方法图 / 结果图 / 结果表
- 论文到底是实验论文还是实证 / 历史 / 综述型论文

## Final QA Checks

写完后回头逐项核对：

- 摘要原文是否实际出现在正文里
- 每条核心结论是否都能追溯到图、表、公式或页码
- 关键数字是否都能在原文中重新找到
- 是否把图贴上去却没有解释它支持什么
- 是否把非实验型论文硬写成实验论文
- 是否说明了证据边界和不确定性
